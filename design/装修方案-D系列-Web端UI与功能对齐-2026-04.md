## 一、总 Definition of Done（与未完成清单 A3 一致）

当 D 系列「装修」完成时，应同时满足：

1. **视觉**：主应用各功能区与 v0 参考 **同族布局与信息层级**，验收口径为 **约 98%** 与规范 § 及组件参考一致（余量见下文「与 v0 的允许差异」）。
2. **功能**：全部接 **真实数据**（非 mock），**空态 / 加载态 / 错误态** 用户可见可理解。
3. **联动**：跨模块链路（见下文各节「关联功能验收」）**可手测走通**，且交互不卡顿。
4. **工程**：`npm run build` 通过；关键路径无明显掉帧。

**完成度表述约定**：


| 维度            | 含义                                                              |
| ------------- | --------------------------------------------------------------- |
| **功能**        | 业务逻辑、存储、API、联动是否在主应用中存在且可用（实用、可测）。                              |
| **UI（相对 v0）** | 结构、密度、三态、组件层级是否与 v0 / 规范 § **同族**；目标 **~98%**，**不**要求与演示稿逐像素一致。 |


下文「**功能：已具备主路径**」指 master-checklist 与代码层面已具备主要能力；「**UI：~98% 同族**」指与 v0 在骨架上一致，**剩余差异** 主要为 **主题 token、间距、动效强弱** 等（如全站背景/纸面色系已随 `@theme` / `index.css` 迭代调整），在发版说明或 PR 截图中备注即可，**不**单列成「与 v0 两条产品线」。

## 二、专业写作平台的共性页面特性（全站）

无论哪一 D，落地页都应体现：

- **心流友好**：主操作路径短、次要操作可折叠；长会话场景下滚动与焦点可预期。
- **数据主权感**：本地优先、云同步、导出、危险操作有确认与说明（与隐私/设置一致）。
- **AI 可解释**：生成/辅助前有门控与成本意识；材料来源（作品/本书锦囊/藏经）在侧栏或摘要中可感知。
- **可恢复**：切章、刷新、失败重试不丢稿；与草稿区、快照、自动保存策略一致。
- **无障碍与键盘**：主要弹层有标题；⌘K、Esc 等与壳层约定一致。

### 与 v0 的允许差异（约 2% 余量）

以下 **不** 视为「未完成 UI 对齐」，而视为 **总体规划内、与 v0 并行的工程/主题选择**：

- **色彩与表面**：`--app-color-*`、`color-mix`、纸面/背景/边框在深浅色下的表现；与昨晚以来在 `index.css` 等处的 **背景色、级联主题** 调整同类。
- **组件实现路径**：主应用以现有 `src/components/ui/*` + Tailwind 为主；v0 中部分 Radix 封装若未逐项引入，以 **同族观感与可达性** 为准，不要求组件名一致。

**D1～D8**：各节 **§6 任务完成度与 v0 目标** 均按上述口径书写（实用功能 + **~98%** 同族）；**不**再单列「未收官」类措辞，除非与规范 **布局/信息架构** 明显不一致。

## D1 — 留白（作品库门户）`/library`

### 1. 定位

**创作入口与作品治理中心**：新建、筛选、打开作品；从这儿进入几乎所有模块。

### 2. 页面应具备的功能


| 能力            | 作用                                                                         |
| ------------- | -------------------------------------------------------------------------- |
| **作品列表**      | 网格/列表、排序、搜索；展示封面、字数、章数、更新时间等 **可决策信息**。                                    |
| **新建 / 编辑作品** | 富表单：标题、**简介**、**状态**、**标签 chip**（与数据模型一致）；创建成功后的跳转策略（如沉浸式 `?zen=1`）符合产品约定。 |
| **单作品操作**     | 继续写作、进推演/本书锦囊（快捷入口）、编辑元数据、删除、封面等（以实际代码为准）。                                 |
| **导入**        | 从文件导入作品（与存储层一致）。                                                           |
| **三态**        | 无作品 **空态**；列表 **加载态**；操作失败 **错误态** 可见。                                     |


### 3. 关联模块与数据流


| 关联                         | 作用                                                      |
| -------------------------- | ------------------------------------------------------- |
| → **写作** `/work/:id`       | 「继续写作」打开正确作品（及章节会话策略）。                                  |
| → **锦囊** `/work/:id/bible` | 作品卡或顶栏「落笔」常落此（原圣经设定护栏；属 **落笔域** 子页）。                    |
| → **推演 / 本书锦囊**            | 快捷入口减少迷路。                                               |
| → **数据层**                  | `Work.description` / `status` / `tags` 与本地库、云库、导入归一化一致。 |
| **新建作品 → 推演**              | 清单要求：新建后 **自动进推演或明确提示**（产品句可配置，但闭环要可感知）。                |


### 4. 专业写作平台特性

- 作品是 **资产**：元数据完整、可检索、可备份。
- **空态** 应引导创建或导入，而非白屏。

### 5. 与 v0 / 规范的对照

- 规范 **§2**；组件参考 `liubai-module` 等。
- **验收目标**：信息与布局与 v0 **约 98% 同族**；页面以 **作品治理与进模块** 等实用能力为主。

### 6. 任务完成度与 v0 目标（自评）


| 维度               | 说明                                                                                                                                                                        |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **功能**           | 列表、搜索、排序、网格/列表、作品表单、导入、封面、进写作等 **主路径已具备**。新建 → 推演提示/跳转以 **产品句与当前实现** 保持一致即可，不单独构成「未完成装修」。                                                                                 |

## D2 — 推演（大纲与逻辑规划）`/logic`

### 1. 定位

**大纲、一致性、分支设想、世界与时间轴** 的规划台；产出要 **能回流到写作与问策**。

### 2. 页面应具备的功能


| 能力                      | 作用                          |
| ----------------------- | --------------------------- |
| **作品 / 章选择**            | 明确「对哪一章做推演」。                |
| **规则扫描 / LLM 扫描**       | 发现设定冲突、逻辑问题；可取消、有错误展示。      |
| **三分支预测**               | 剧情分支设想；结果可 **写入写作侧栏草稿**。    |
| **去问策跟进**               | 将分支内容 **预填到问策** 新会话，继续策略对话。 |
| **文策日志**（若产品要求）         | 与「改纲流水」边界见决策记录；**不得做空壳按钮**。 |
| **时间线 / 世界观 / 地图与地点事件** | 与本书锦囊数据联动；列表与编辑信息密度可读。      |
| **三态**                  | 无作品、无章节、无扫描结果时的引导。          |


### 3. 关联模块与数据流


| 关联               | 作用                              |
| ---------------- | ------------------------------- |
| → **AiPanel 草稿** | `writeAiPanelDraft` 等，写作页合并入正文。 |
| → **问策**         | `navigate` + state 预填会话。        |
| ← **本书锦囊 / 作品**  | `repo` 读人物、时间线、地点等。             |


### 4. 专业写作平台特性

- 推演是 **决策沙盒**：输出必须 **可追溯、可落地**（草稿/会话），而不是一次性聊天。

### 5. 与 v0 / 规范的对照

- 规范 **§3**；三栏信息架构与 v0 **约 98%** 同族对齐。

### 6. 任务完成度与 v0 目标（自评）

## D5 — 本书正文（写作）+ 锦囊 + 概要：`/work/:id`、`/work/:id/bible`、`/work/:id/summary`

> **命名（2026-04-11）**：**「写作」** = `/work/:id` 正文页（**留白新建成功默认进入**）；**「锦囊」** = 原圣经设定护栏 `/bible`；**「落笔」** = 七模块名与 §5 总域（**≠** 锦囊、**≠** 写作页名）。与 spec §5 / §9 一致。

### 1. 定位

**写作页** = 正文创作的核心；**锦囊页** = 设定护栏（顶栏「落笔」当前常链至此子域）；**概要** = 成稿辅助。

### 2. 页面应具备的功能


| 能力       | 作用                                                   |
| -------- | ---------------------------------------------------- |
| **写作页**  | 章节树、正文编辑、保存态、字数、查找替换、导出等；**切章不串稿**。                  |
| **右栏**   | AI / 概要 / 本书锦囊 / 参考；**草稿区与合并入正文**（diff 确认）；材料与装配器一致。 |
| **沉浸写作** | Zen；Esc 与壳层一致；退出后焦点回编辑器。                             |
