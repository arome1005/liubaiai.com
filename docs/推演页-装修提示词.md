## 推演页（/logic）装修提示词（复制即用）

你是本仓库的编码助手。现在执行 **推演功能页 `/logic` 的 UI 装修**：把主应用中已实现的功能保持不变，把界面与信息架构对齐到 v0（约 98% 同族），并能用真实数据手测。

### 0) 真源（必须先打开看过再写）

- **规范**：`design/v0-ui-reference/docs/ui-design-specification.md` 的 §<填写>
- **v0 模块**：`design/v0-ui-reference/components/modules/<填写：logic-module 或同名文件>`（以此为真源）
- **对照截图**：<粘贴/附图：推演页 v0 UI 截图>

### 1) 硬约束（必须遵守）

- **功能优先**：不删、不改坏既有业务逻辑、存储、路由、API、权限、联动。装修主要是 JSX 结构、className、样式、组件层级。
- **逻辑改动边界**：若必须动逻辑，仅限为适配布局所必需的最小改动，并在最终回复里用一句话说明原因。
- **范围克制**：只改 `/logic` 页面及其直接相关的组件/样式文件；禁止顺手大重构、禁止改无关页面/文档/schema。
- **真实数据**：不得用 mock 替换真实数据；已有功能必须可手测。
- **未完成能力**：对仓库中“未实现/待开发”的功能，只能做 UI 占位（disabled / coming soon / skeleton），不得伪造逻辑。
- **完成后自检**：必须跑 `npm run build` 并修到通过。

### 2) 允许改动范围（只允许这些）

- JSX 结构、组件层级、布局（分栏/工具栏/卡片/列表/Tab）、className、样式、交互骨架（按钮位置/可点击区域/悬浮层/空态）
- 为对齐布局所需的最小 state/props 调整（不得改变数据含义，不得改存储结构）

### 3) 你开始动手前必须先输出（一句话）

用一句话复述本次任务，格式固定为：

> 我在装修 `**/logic` 推演页**，参考 v0 的 `**<v0 模块路径>`** 与规范 **§****，保持现有功能与数据流不变。**

### 4) 你必须按这个顺序执行

1. 打开并阅读 v0 真源（规范 §n + module 文件），摘出关键布局骨架（区块顺序/分栏/密度/交互入口）。
2. 在主应用中仅做“换皮对齐”的最小 diff（不碰业务）。
3. 保证深色/浅色都可读、不糊边（在现有 token 下调整）。
4. 运行 `npm run build`，修复因装修引入的错误。

### 5) 未完成项呈现策略（默认）

如果 v0 有入口但主应用未实现：

- **默认**：保留入口位置与外观，但设置为 disabled，并显示 “待开发”/“即将上线”（tooltip 或次级文案），不触发实际逻辑。

### 6) 最终回复格式（必须包含）

- **改了哪些文件**：列出文件路径
- **对齐了哪些 v0 点**：3–8 条（布局/层级/密度/交互骨架）
- **约 2% 允许差异**：落在哪些 token（背景/纸色/边框/圆角/阴影/间距/现有组件等价实现）
- **是否改了逻辑**：
  - 若否：一句话确认“功能与数据流不变”
  - 若是：一句话说明“为什么不得不动 + 动了哪里（最小）”
- **自检结果**：`npm run build` 是否通过

### 7) 推演页核对清单（装修时别丢）

- **入口与导航**：从 `/library`/其它入口到 `/logic` 的跳转不变；返回/切换不丢状态（若原来就不丢）。
- **数据源**：作品/章节/推演结果/引用资料等读取路径不变。
- **写回**：任何“保存/应用/生成/采纳”按钮的写回逻辑不改（仅换 UI）。
- **错误与空态**：无作品/无章节/无推演结果/加载中/失败态在 v0 骨架下仍可理解。
