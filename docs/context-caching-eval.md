# Context Caching 评估（§11 步 28）

> **结论**：当前主应用为 **浏览器直连厂商 API**（`src/ai/providers.ts`），**无**统一服务端持有长 system 前缀；**Prompt / Context Caching**（OpenAI、Anthropic、Gemini 等）多在 **服务端或官方 SDK** 侧按 **稳定前缀 + TTL** 计费优化。  
> **一期建议**：**不单独做 POC 阻塞发布**；若后续有 **Node 代理层** 或 **固定超长 system**，再按厂商文档对 **前缀哈希、失效条件、按 token 计费** 做实测。

## 1. 机制摘要（产品视角）

- **适用**：重复发送 **相同前缀**（system、长本书锦囊导出、工具定义）且厂商支持 **cache read / cache write** 计费。
- **失效**：模型/前缀/参数变更会导致 cache miss；需 **监控命中率** 才有意义。
- **与本项目**：写作侧栏已做 **上下文截断与隐私门控**（`assemble-context.md`）；长内容主要来自 **全书锦囊导出、邻章概要、RAG 片段** — 若未来在 **服务端** 合并为单一前缀，才值得评估 cache。

## 2. 与本仓库的对应关系


| 维度    | 现状                             | 若要做缓存                                  |
| ----- | ------------------------------ | -------------------------------------- |
| 调用路径  | 浏览器 `fetch` / SSE              | 需 **服务端** 或 **官方 SDK** 的 cache 头/参数    |
| 前缀稳定性 | 用户随时改本书锦囊/章/设置                   | 需定义 **版本号** 或 **哈希** 作为 cache key 的一部分 |
| 成本    | 已用粗估 token（`approx-tokens.ts`） | 对比 **cache read / write** 账单与 miss 率   |


## 3. 建议下一步（非阻塞）

1. 在 **后端模型配置** 或 **发布说明** 中记录：若月 token 超过阈值，再立项 **代理 + 缓存**。
2. 若引入 **OpenAI/Anthropic** 官方缓存 API：在 **单环境** 对 **同一作品** 连续两次请求做 **A/B 对比**（延迟、账单字段），再写数据到 `docs/性能说明.md`。
