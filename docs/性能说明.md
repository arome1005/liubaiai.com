# 性能预期（留白写作）

## 目标场景

- **章节数**：单作品数百章级（如 500+）仍可操作侧栏滚动、拖拽排序、切换章节；全书字数统计优先使用章内 `**wordCountCache`**，避免对全部正文重复扫描。
- **单章正文**：数万～十万汉字级时，以纯文本 `textarea` 编辑，依赖浏览器布局；若未来换富文本或分块编辑器，需单独压测。

## 当前实现要点

- 全书搜索为**字面量线性扫描**各章正文，章数极多且单章极大时可能卡顿；可后续改为 Web Worker 或索引（SQLite FTS 等）。
- IndexedDB 通过 Dexie 访问；备份 zip 为一次性序列化，库很大时导出/导入会占用内存与主线程时间。
- **参考库（3.7）**：阅读器仅通过 `**[refWorkId+ordinal]`** 按需加载当前段与相邻段正文，**不**再一次性 `listReferenceChunks` 全表进内存；无分词时的**字面回退检索**对全库扫描的分块数有上限（见 `writing-store-indexeddb.ts` 中常量），避免极端情况下拖垮主线程。**参考全文检索**命中的是正文分块，**不**按写作进度裁剪；与 2.6 游标一致的防剧透依赖摘录「关联创作章节」+ 侧栏过滤（见路线图 3.8）。

## 抽样

- 尚未建立自动化基准；迭代时在上述量级下人工抽查「切换章、搜索、导出」是否可接受即可。

## 手动压测脚本（建议发布前跑一遍）

以下脚本目标是快速发现“必现卡死/内存暴涨/保存异常”，不追求精确毫秒指标。

### A. 创作库

1. **打开写作页**：进入任意作品的 `/work/:workId`，连续切换 10 次章节，观察是否出现明显卡顿与保存异常提示。
2. **大章输入与保存**：在一章中连续粘贴/输入一段较长文本（例如 1～2 万字），等待自动保存指示稳定后刷新页面，确认内容仍在。
3. **章内查找/替换**：用常见关键词做 3 次查找、1 次替换，确认 UI 不阻塞。
4. **全书搜索**：在“全书搜索”中用一个高频词触发多命中，确认结果列表可滚动、跳转可用。
5. **导入旧书**：分别用 `.txt` / `.md` / `.docx` 导入一本旧书（建议 50+ 章），确认自动识别章节标题与正文大致一致。
6. **备份/恢复**：设置页导出 zip → 清空站点数据（或新开浏览器 profile）→ 从 zip 恢复，抽样对比章数与正文。

### B. 参考库（原作资料）

在「单书 ≥ 百万字、分块数百段」的样本上抽查：

1. **阅读器**：打开书目后切换「上一段 / 下一段」，确认仅当前段与相邻段加载，无长时间卡顿。
2. **全文检索**：有关键词命中时结果可跳转高亮；无分词时的字面回退在极大库上仍应在可接受时间内返回（回退路径有扫块上限）。
3. **导入**：大文件导入时进度与索引阶段可完成；若 `.txt` 非 UTF-8，应看到编码提示并可选择取消。

### C. AI（可选）

1. **Ollama**：选择 Ollama，生成一次，确认“草稿区出结果”“取消/重试可用”。
2. **云端 AI 防误传**：未在设置中勾选同意与启用时，选择 OpenAI/Claude/Gemini 应显示禁用提示，点击生成不会发送请求。
3. **上传范围开关**：启用云端后，关闭“正文/圣经/摘要”等开关，确认对应内容不会被注入（以注入预览为准）。

### 参考库（3.7）手动验收清单

建议在「单书 ≥ 百万字、分块数百段」的样本上抽查：

1. **阅读器**：打开书目后切换「上一段 / 下一段」，确认仅当前段与相邻段加载，无长时间卡顿。
2. **全文检索**：有关键词命中时结果可跳转高亮；无分词时的字面回退在极大库上仍应在可接受时间内返回（回退路径有扫块上限）。
3. **导入**：大文件导入时进度与索引阶段可完成；若 `.txt` 非 UTF-8，应看到编码提示并可选择取消。

## 构建与主包体积（§11 步 49 · Living）

- `**npm run build`**：须保持通过（`tsc -b && vite build`）。
- **主 JS chunk（单文件）**：当前构建产物约为 **2.75 MB（gzip ≈ 801 KB）**；Vite 会提示 `>500 kB` 告警，属预期。后续可通过路由级 `import()`、拆 mammoth 与主包等收敛（与 `design/master-checklist.md` §D.4 一致）。
- **其他大块**：`pdf.worker.min*.mjs`（PDF 解析）体积独立列出，首屏不阻塞主逻辑。
- **已知构建提示**：`mammoth` 同时被静态与动态引用时会出现 `INEFFECTIVE_DYNAMIC_IMPORT` 提示；不影响发布，与动态拆 chunk 优化可择机同 PR。

### 关键路径抽样（可复现 smoke，步 49）

本地 `npm run dev` 或 `npm run preview` 后，在**同一浏览器会话**内按序抽查（不要求秒表，以「无明显卡死、可完成主操作」为准）：


| 页面       | 路径              | 建议操作                              |
| -------- | --------------- | --------------------------------- |
| 首页 / Hub | `/`             | 点击至少一个模块入口（如作品库、藏经）；若有「继续创作」可点一次。 |
| 作品库      | `/library`      | 打开「新建作品」弹窗再取消，或进入一部已有作品的写作页。      |
| 写作       | `/work/:workId` | 切换章节一次；侧栏打开 AI 再收起（不必真请求模型）。      |
| 藏经       | `/reference`    | 若已有书目：打开一本、翻一段；无书目则确认空态与导入说明可见。   |
| 设置       | `/settings`     | 滚动到「数据」或「隐私」区块，确认锚点/分区可访问。        |


### 大文件 / 弱网（可选）

- **写作页**：极长单章（数万字级）首次打开与输入时，以本机为准观察；已知瓶颈主要在浏览器布局与自动保存频率，而非单次 HTTP。
- **参考库导入**：超大 `.txt` 导入时主线程会忙于分块与建索引，属预期；弱网下仅影响「从磁盘读入」速度，不涉及上传参考书正文。
