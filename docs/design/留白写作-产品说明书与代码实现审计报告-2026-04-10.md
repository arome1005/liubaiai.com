## 六、开发建议

### A. 架构层面

1. **后端 API 覆盖率严重不足**
  - 当前后端只覆盖作品/章节/认证，约占数据模型的 20%
  - 建议按模块批量补充 REST API：bible → wence → sheng-hui-snapshots
  - 或考虑将更多操作走 Supabase 直连（配合 RLS），减少后端编写量
2. **推演模块路由整合**
  - `/logic` 与 `/v0/tuiyan` 并存是最大混乱点
  - 建议：将 `/v0/tuiyan` 页面直接注册为 `/logic`，废弃旧 `/logic` 的地点导图（地点功能合并到推演右侧或子 Tab）
3. **跨模块写回总线缺失**
  - 各模块写回是散点实现，缺乏统一的 "写回事件总线"
  - 建议建立 `WritingActionBus` 或 Context，统一处理：草稿写入、合并确认、失败恢复

### B. 产品层面

1. **生辉是最大差距模块**
  - 说明书定义的差异化功能（上下文装配/版本对比/细纲驱动生成）几乎全部缺失
  - 建议作为 P1 专项 Sprint 重点攻坚
2. **成本门控是合规风险**
  - 用户可能无感知地触发大量 AI 调用
  - 建议 P1 补充：token 粗估展示 + 超阈值确认弹窗，保护用户
3. **全局作品上下文一致性**
  - 当前状态：进入写作页有上下文，离开后其他模块不确定是否携带
  - 建议：强化 `lastWorkId` 在 AppShell 顶栏的持续展示，形成作品锚点感

### C. 工程质量

1. **缺少端到端测试**
  - 仅见 `import-work.test.ts`，四条核心链路没有测试覆盖
  - 建议：为每条核心链路（推演→写作、流光→写作、RAG→写作）写 E2E 测试
2. **localStorage Key 命名空间管理**
  - `liubai:`* key 数量已较多（12+ 个），缺少统一清单
  - 建议：在 `src/util/storage-keys.ts` 统一枚举所有 localStorage key
3. **IndexedDB Schema 版本管理**
  - 当前 `SCHEMA_VERSION=27`，升级路径需要仔细维护
  - 建议：为每个 migration 版本写明变更说明，防止线上用户数据损坏
4. **错误边界覆盖不完整**
  - 仅顶层有 `ErrorBoundary`，各模块页面无独立边界
    - 建议：每个大模块页面（EditorPage/BiblePage/ChatPage）添加独立 ErrorBoundary
