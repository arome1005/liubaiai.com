# 留白写作 — 总体规划改进版（2026-04）

> **定位**：在 `docs/总体规划-路线图与导航整合.md`（改版 **v2**）与 `docs/路线图.md` 之上，吸收 **当前仓库真实进度** 与 **产品目标修订**（功能按 §11 全集落地 + **主应用 UI 与 v0 参考极高相似度**）。  
> **用途**：全员对齐「做到哪、还差什么、先改什么」；**在整体未完工前**主动调整架构与清单，避免「全部功能堆完再换皮」导致的一次性大爆炸与不可控缺陷。  
> **真源分工**：**执行划项**以 `design/master-checklist.md` 为准（**§B** §11 步 **1～54**、**§E** UI‑V0 二轮、**§F** 深度改进对标）；**视觉规范**以 `design/v0-ui-reference/docs/ui-design-specification.md` 与 v2.0 参考工程为准；**规划级缺陷 / 分期**以 `docs/bug-backlog.md` 为准（与 §D.4 互补）；**产品深化命题**以 `docs/留白写作-深度改进建议书v1.md` 为论述真源，**落地划项**以清单 **§F** 为准（与 §B 并行，不替代 §11 闭项）。

## 1. 为什么现在就要改规划（与你的担心对齐）

- **功能与 UI 强耦合**：同一页上，状态一多（空态 / 加载 / 错误 / 长列表 / 侧栏），若末期才从「手写 CSS 渐进换皮」切到 **与 v0 同构的组件与 token**，极易出现 **布局错位、回归遗漏、交互分叉**，表现为「无数大 bug」集中爆发。  
- **正确节奏**：**能力**继续按 §11 与清单迭代；**表现层**用单独一条 **UI‑V0 收敛轨** 与功能 **并行、同 PR 可交织**，先 **基建（栈与 token）** 再 **逐页迁移**，每页保留 **真实 `repo` / 路由 / AI**。  
- **目标表述**：不承诺 **100%** 像素一致；约定 **约 96%～98%** 视觉与结构一致，余量为字体渲染、极端数据、第三方编辑器嵌套等 **可接受微调**，**不改变整体信息架构与版式**。

## 6. UI‑V0 渲染收敛轨（新建，与功能轨并行）

> 详见 `design/master-checklist.md` **§E**。原则摘要：

1. **唯一上线构建**：仍为仓库根目录 **Vite 主应用**；业务逻辑不迁往 Next 子工程。
2. **视觉真源**：`design/v0-ui-reference` + `ui-design-specification.md`；实现上 **主工程接入与 v0 同构的样式/组件体系**（具体技术选型在 §E.1 落地时写死，例如 Tailwind v4 + shadcn 类组件）。
3. **迁移顺序建议**：全局 token → **AppShell** → **作品库 / Hub** → **写作壳 + 侧栏** → 设置 / 藏经 / 圣经 / 概要 → 推演 / 流光 / 问策 → **生辉（功能就绪后同轨套皮）**。
4. **每步验收**：`npm run build`；该页 **原有功能路径可复现**；与规范 **对照走查**（空态/错误/长标题至少各看一眼）。
5. **禁止**：从子工程 **直接 import** 源码进 Vite（双栈不兼容）；应 **复制结构并接 `repo`**。

## 7. 风险与门禁（防止「改规划」本身引入灾难）

- **双轨并行**：同一时期既有 §11 功能 PR，也有 §E UI 迁移 PR 时，**优先保证行为不退化**；UI PR 必须带 **手动回归路径**（或后续补 E2E 时再固化）。  
- **大文件/编辑器**：CodeMirror 与 shadcn 混排处单独列 **接缝检查项**（焦点、滚动、移动端若适用）。  
- **设计漂移**：v0 参考若再导出新版，需记录 **对照版本**（日期或目录快照说明），避免 silently 追新。  
- **清单Living**：任何「规划修订」落地后，**同步改 `master-checklist.md` + 本改进版日期/小节**，避免口头约定；**任务清单记录是否完整**以 **§9** 审计表为准，大版本合入前扫一眼。
